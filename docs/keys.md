# Ключи капчи: получение и безопасность

Читайте этот файл перед первой интеграцией и при работе с секретами: здесь описаны оба ключа капчи, где их взять (консоль, CLI, API) и как обращаться с ключом сервера, чтобы не скомпрометировать защиту.

## Два ключа и их роли

При создании капчи автоматически генерируется пара ключей. Они генерируются один раз и **никогда не меняются** — ротация возможна только созданием новой капчи.

| Ключ | Роль | Где используется | Публичность |
|---|---|---|---|
| **Ключ клиента** (`client_key`) | Отрисовка виджета | Фронтенд: `data-sitekey` в автоматическом методе, параметр `sitekey` в `render()` и в React-компоненте | Публичный — виден в HTML страницы |
| **Ключ сервера** (`server_key`) | Получение результата проверки | Бэкенд: параметр `secret` в POST-запросе к `https://smartcaptcha.yandexcloud.net/validate` | Приватный — секрет |

Третий идентификатор — одноразовый токен: его генерирует сам сервис после проверки пользователя, он действителен 5 минут и передается с фронтенда на бэкенд. Жизненный цикл токена — в [overview.md](overview.md), проверка — в [server-validation.md](server-validation.md).

### Формат ключей

Ключ клиента всегда имеет префикс `ysc1_`, ключ сервера — `ysc2_`. Следующие 20 символов у них совпадают — по этому признаку удобно убедиться, что ключи из одной пары:

* `client_key`: `ysc1_yzF85********`
* `server_key`: `ysc2_yzF85********`

Если виджет отрисовывается, а `/validate` отвечает `failed` — проверьте, что ключи из одной пары (одной капчи) и что в `secret` не попал ключ клиента.

## Где взять ключи

### Консоль управления

1. В [консоли управления](https://console.yandex.cloud) выберите каталог.
2. Перейдите в сервис **SmartCaptcha**.
3. Нажмите на имя капчи (или создайте новую — см. [captcha-management.md](captcha-management.md)).
4. На вкладке **Обзор** скопируйте значения полей **Ключ клиента** и **Ключ сервера**.

### CLI

Ключ сервера возвращает команда `get-secret-key` (нужен идентификатор или имя капчи):

```bash
# Получить ключ сервера капчи; подробности — в docs/keys.md этого скилла
yc smartcaptcha captcha get-secret-key <идентификатор_капчи>
```

Вместо позиционного аргумента можно использовать флаги `--id <идентификатор>` или `--name <имя>` (имя резолвится в id в текущем каталоге). Список капч и их id: `yc smartcaptcha captcha list`. Остальные CLI-команды — в [../references/management-api.md](../references/management-api.md).

### API

* REST — метод `Captcha.GetSecretKey`:

  ```
  GET https://smartcaptcha.api.cloud.yandex.net/smartcaptcha/v1/captchas/{captchaId}:getSecretKey
  ```

  Ответ: `{"serverKey": "string"}`.
* gRPC — вызов `CaptchaService/GetSecretKey`.

Такое получение ключа по API удобно для CI/CD и автоматизации: секрет не хранится в коде, а запрашивается по IAM-авторизации в момент деплоя.

## Безопасность ключа сервера

Ключ сервера подтверждает, что запрос к `/validate` пришел от владельца капчи. Тот, у кого он есть, может валидировать токены вашей капчи — то есть выдавать ботов за проверенных пользователей. Правила:

* **Не публикуйте во фронтенде.** Ключ сервера используется только на бэкенде. Если он попал в HTML/JS-бандл — он скомпрометирован.
* **Не коммитьте в репозиторий.** Не храните ключ в коде и конфигах под версионным контролем — читайте его из переменной окружения (например, `SMARTCAPTCHA_SERVER_KEY`) или из секретницы. Официальная рекомендация — [Yandex Lockbox](https://yandex.cloud/ru/docs/lockbox).
* **Не пересылайте и не храните в открытом виде.** Никогда не пересылайте ключ сервера (мессенджеры, тикеты, логи) и не храните в незашифрованном виде; не допускайте попадания в открытый доступ.
* **Плейсхолдеры в примерах.** В шаблонах этого скилла ключи обозначены `<ключ_клиента>` и `<ключ_сервера>` — подставляйте реальные значения только через окружение/секретницу, не в текст файлов.

### Если ключ скомпрометирован

Ключи капчи не меняются, поэтому единственный способ ротации: **создайте новую капчу и замените ей старую** (см. [captcha-management.md](captcha-management.md)). Затем обновите ключ клиента на фронтенде и ключ сервера в окружении бэкенда; старую капчу удалите.

## Ключ клиента и домены

Ключ клиента публичный по определению — скрыть его нельзя. От использования вашего ключа на чужих сайтах защищает валидация домена: по умолчанию капча работает только на сайтах из списка, указанного в ее настройках. Если вы отключили проверку домена, кто угодно сможет использовать ваш ключ — тогда обязательно проверяйте поле `host` в ответе `/validate` на своем сервере и отклоняйте результаты с неожиданных доменов. Подробнее — в [overview.md](overview.md) (раздел «Валидация домена») и [captcha-settings.md](captcha-settings.md).
