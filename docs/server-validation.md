# Серверная проверка токена (/validate)

Читай этот файл, когда реализуешь backend-часть интеграции SmartCaptcha: приём токена из формы и его проверку через API `/validate`. Полный справочник параметров и ответов — в [../references/validate-api.md](../references/validate-api.md).

## Зачем проверять токен на сервере

Виджет капчи работает на стороне клиента и лишь выдаёт токен — сам по себе он ничего не защищает. Токен добавляется в HTML-страницу как скрытое поле внутри контейнера виджета:

```html
<div id="captcha-container" class="smart-captcha" ...>
    <input type="hidden" name="smart-token" value="<токен>">
    ...
</div>
```

Любое поле формы клиент может подделать: бот отправит форму без токена, с выдуманным или уже использованным токеном. Поэтому результат проверки нельзя брать из клиентского кода — его запрашивают только с сервера, POST-запросом к `/validate` с секретным ключом сервера. Ключ сервера никогда не попадает на клиент (см. [keys.md](keys.md)). Без серверной проверки капча не выполняет свою функцию.

## Запрос к /validate

Отправь POST-запрос на `https://smartcaptcha.yandexcloud.net/validate`, передав параметры в формате `x-www-form-urlencoded`:

```
secret=<ключ_сервера>&token=<токен>&ip=<IP-адрес_пользователя>
```

* `secret` — ключ сервера (см. [keys.md](keys.md)).
* `token` — одноразовый токен, полученный после прохождения проверки. При автоматическом подключении бери его из поля формы `smart-token` ([widget-auto.md](widget-auto.md)); при расширенном методе токен приходит в колбэк ([widget-advanced.md](widget-advanced.md)).
* `ip` — IP-адрес пользователя, с которого пришёл запрос. Параметр не обязателен, но документация просит его передавать: это помогает улучшить качество работы SmartCaptcha.

Минимальный пример (cURL):

```bash
curl -X POST https://smartcaptcha.yandexcloud.net/validate \
  -d "secret=<ключ_сервера>" \
  -d "token=<токен_из_формы>" \
  -d "ip=<IP_пользователя>"
```

### Откуда брать IP за прокси

Способ получения IP-адреса пользователя зависит от фреймворка и прокси. Если приложение стоит напрямую в интернете, бери адрес соединения (например, во Flask это `request.remote_addr`). Если перед приложением reverse-proxy или балансировщик, адрес соединения — это IP прокси, а не пользователя: бери реальный IP из заголовка, который проставляет ваш прокси (обычно `X-Forwarded-For` или аналог), и доверяй этому заголовку только от своего прокси. Если достоверный IP получить нельзя — лучше не передавать `ip` вовсе, чем передать IP прокси.

## Ответ сервиса

Сервис возвращает JSON с полями `status` и `message`; при `status: ok` добавляется поле `host` — сайт, на котором была пройдена проверка. Основные варианты:

* `{"status": "ok", "message": "", "host": "example.com"}` — это человек, проверка пройдена на `example.com` (порт, если нестандартный, входит в `host`: `example.com:8080`).
* `{"status": "ok", "message": "", "host": ""}` — пустой `host` может означать, что облако заблокировано или произошёл внутренний сбой сервиса.
* `{"status": "failed", "message": ""}` — это робот.
* `{"status": "failed", "message": "Invalid or expired Token."}` — поддельный, повреждённый, просроченный или уже использованный токен. Это робот.
* `{"status": "failed", "message": "Authentication failed. Secret has not provided."}` — запрос без ключа сервера.

Полная таблица ответов с примерами — в [../references/validate-api.md](../references/validate-api.md).

Если при отрисовке виджета передавались [метаданные](./captcha-settings.md) (`metadata` в `render()`), по официальной схеме проверки ответ `/validate` содержит метаданные из справки о прохождении капчи — сверь их со значениями, сохранёнными на бэкенде при выдаче.

## Ориентируйся только на поле status

Для обработки ответа используй только `status`:

* `ok` — запрос обработан успешно, пользователя можно пропускать.
* `failed` — произошла ошибка.

Ошибки делятся на два типа:

* **Это робот** — `"status": "failed"`, поле `message` пустое.
* **Ошибка в запросе** — `"status": "failed"`, `message` содержит описание ошибки (недействительный токен, отсутствует ключ сервера и т. п.). Такие ошибки выявляй и исправляй на этапе разработки и тестирования — в проде их быть не должно.

> **Важно.** Поле `message` не предназначено для обработки в коде с помощью условий или сравнений — его текст может меняться. Используй `message` только для диагностики и логов.

## Одноразовость и срок жизни токена

> **Важно.**
> * Токен действителен 5 минут. По истечении этого времени он становится недействительным, и пользователю нужно снова пройти проверку.
> * Токен можно использовать только один раз. Повторная валидация того же токена вернёт `"status": "failed"` с сообщением `Invalid or expired Token`.

Практические следствия:

* Проверяй токен сразу при получении формы, а не отложенно (очереди, повторные обработчики) — иначе он истечёт.
* Не кешируй и не переиспользуй токен: одна отправка формы — одна проверка. Ретрай запроса к `/validate` с тем же токеном после уже полученного ответа даст `failed`.
* Если пользователь долго заполняет форму, токен может истечь до отправки — предусмотри в UI повторное прохождение капчи (см. [troubleshooting.md](troubleshooting.md)).

## Fail-open при недоступности сервиса

> **Важно.** Ошибки HTTP-протокола (код ответа не 200), сетевые ошибки и таймауты рекомендуется обрабатывать как ответ `"status": "ok"` — то есть пропускать пользователя.

Почему так: капча — фильтр от ботов, а не механизм аутентификации. Если при сбое SmartCaptcha отвечать отказом (fail-close), недоступность внешнего сервиса заблокирует всех живых пользователей и «положит» вашу форму. Официальная рекомендация — обрабатывать код ответа не 200 как `ok`, чтобы при обработке запроса пользователя не было задержки. Fail-open обрабатывай так во всех ветках: HTTP-код != 200, ошибка соединения, таймаут, невалидный JSON в ответе.

## Таймауты

Ставь короткий таймаут на запрос к `/validate`, чтобы сбой сервиса не подвешивал обработку формы. Разграничь два факта, чтобы не смешивать их в комментариях к коду:

* **fail-open — рекомендация документации** (см. выше);
* **конкретное значение таймаута документация не рекомендует.** 1 секунда — это значение из официальных примеров кода (Python `timeout=1`, PHP `CURLOPT_TIMEOUT 1`), а не норматив. Выбирай сам: агрессивный таймаут учащает fail-open (капча чаще «пропускает всех»), длинный — задерживает пользователей при сбое.

Таймаут-бюджет должен покрывать весь запрос: соединение + получение заголовков + чтение тела ответа. Истечение таймаута обрабатывай по стратегии fail-open.

## Конвенции скилла (не требования документации, но следуй им)

* **Нет токена в запросе → сразу 403, без похода в `/validate`.** Отсутствующий/пустой `smart-token` — это отказ клиента пройти проверку, а не сбой сервиса: fail-open здесь неуместен (бот обошёл бы капчу, просто не прислав токен), а лишний запрос расходует квоту.
* **Не задан ключ сервера → громкая ошибка конфигурации** (исключение при старте/первом вызове), а не тихий фолбэк на плейсхолдер: с неверным `secret` сервис отвечает `failed` всем — молчаливый отказ всем живым пользователям, который сложно заметить.
* **Ответ приложения при непройденной капче — HTTP 403** с машиночитаемым телом (например, `{"ok": false, "error": "captcha_failed"}`).
* **Транспорт токена**: классическая форма — скрытое поле `smart-token`; JSON API — поле `smart-token` в теле; middleware/заголовки — `X-Smart-Token`. В middleware не «съедай» тело запроса при извлечении токена — восстанови его для нижестоящего обработчика.

## Асинхронные рантаймы (event loop)

В неблокирующих рантаймах PHP (AMPHP, ReactPHP, Swoole) и любых event-loop окружениях нельзя вызывать блокирующие клиенты (`curl_exec`, `file_get_contents`) — они останавливают весь процесс. Используй нативный async HTTP-клиент рантайма; для AMPHP v3 есть готовый шаблон [../templates/validate-amphp.php](../templates/validate-amphp.php). Требования те же + два специфичных:

* один таймаут-бюджет (например, `TimeoutCancellation`) должен покрывать и запрос, и чтение тела ответа;
* логируй fail-open неблокирующим логгером рантайма (для AMPHP — `amphp/log`), а не синхронной записью в файл.

## Локальное тестирование без реального сервиса

Для юнит- и смоук-тестов не ходи в реальный `/validate` (токены одноразовые, нужен живой виджет). Подними локальный мок — готовый роутер-скрипт [../templates/mock-validate.php](../templates/mock-validate.php) (`php -S 127.0.0.1:8099 mock-validate.php`) эмулирует канонические сценарии: `ok`, робот, просроченный токен, запрос без секрета, HTTP 500, битый JSON, таймаут. Передай в свой код тестовый URL вместо боевого и прогони все ветки, включая fail-open.
